Design Pattern

As we proceed on development we have realised that following a certain design pattern makes work easier and efficient when there are multuple developers. The main design pattern we try to follow MVC. It is perhaps the most common design pattern in the development world. By definition The Model-View-Controller (MVC) is an architectural pattern that separates an application into three main logical components: the model, the view, and the controller. Each of these components are built to handle specific development aspects of an application.

Our Model,View and Controller are as followig:

  • Model:Category,Payment Type, all the transactions
  • View:The UI
  • Controller: Manager Classes

    • In short about these three parts, The Model component corresponds to all the data-related logic that the user works with. This can represent either the data that is being transferred between the View and Controller components or any other business logic-related data. The View component is used for all the UI logic of the application. For example, the Customer view will include all the UI components such as category lists, entries etc. that the final user interacts with. Controllers act as an interface between Model and View components to process all the business logic and incoming requests, manipulate data using the Model component and interact with the Views to render the final output.

    The main reasons behind chosing this design pattern is because separating implementation into separate classes that aren't strictly tied to each other, so if you need to change the code in one area of functionality, you can do it more easily without disturbing other unrelated code. MVC is a good way to achieve this goal in web application architecture. By separating controllers (handling requests and mapping to business logic), models (implementing business logic), and views (presenting and formatting output), it's easier to change code in each of these layers. In one word you can reuse code more easily. It makes code agile, scalable, robust and modular.

    Coding Convention

    By Definition,Coding conventions are a set of guidelines for a specific programming language that recommend programming style, practices, and methods for each aspect of a program written in that language. These conventions usually cover file organization, indentation, comments, declarations, statements, white space, naming conventions, programming practices, programming principles, programming rules of thumb, architectural best practices, etc. These are guidelines for software structural quality. Software programmers are highly recommended to follow these guidelines to help improve the readability of their source code and make software maintenance easier. Coding conventions are only applicable to the human maintainers and peer reviewers of a software project. Conventions may be formalized in a documented set of rules that an entire team or company follows, or may be as informal as the habitual coding practices of an individual. Coding conventions are not enforced by compilers.

    However, ours is a small project and thus all the conventions are not required. Although we tried so far to follow as much as possible. It helped greatly keeping things organised.

    Following are some rules we were stick to:

    • Applying commonly accepted rules
    • Organizing similar classes in a similar way (e.g. the use of attributes and methods in our Fragments or if you look at the classes Entry, Category and PaymentMethod on one side and their manager classes on the other side)
    • The IDs of xml objects are more or less organized in a uniform way
    • XML objects in Fragments are always declared as attributes of the Fragment class and named in a more or less uniform way

      • We are working with Android Studio, It is an amazing IDE because along with other thing it helps to organise codes and designs. Perhaps we couldn't follow the convention all the time but we always kept it in our mind to follow it
